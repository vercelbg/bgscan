# تنظیمات عمومی

فایل پیکربندی: `settings/general_settings.toml`

این فایل رفتار اصلی اسکنر را کنترل می‌کند — از محدودیت‌های اسکن و ترتیب IP‌ها گرفته تا استراتژی اجرای پایپ‌لاین و مدیریت حافظه.

## کنترل اسکن

این تنظیمات مشخص می‌کنند چه تعداد IP اسکن شود، به چه ترتیبی، و چه زمانی متوقف شود.

### `status_interval`

```toml
status_interval = 1000
```

فاصله زمانی بین به‌روزرسانی‌های وضعیت ارسال‌شده به رابط کاربری، بر حسب **میلی‌ثانیه**.

| مقدار | رفتار |
|-------|-------|
| `1000` | هر ۱ ثانیه یک‌بار (پیش‌فرض) |
| `500` | دو بار در ثانیه |

---

### `stop_after_found`

```toml
stop_after_found = 0
```

پس از این تعداد **نتیجه موفق** (IP‌هایی که تمام شرایط اسکن را رد کرده‌اند)، اسکن متوقف می‌شود.

| مقدار | رفتار |
|-------|-------|
| `0` | غیرفعال — همه IP‌ها اسکن می‌شوند |
| `N` | پس از N نتیجه موفق متوقف می‌شود |

> **نکته:** در اسکن زنجیره‌ای (chain-mode)، این محدودیت **فقط برای مرحله آخر** اعمال می‌شود. پشتیبانی چند مرحله‌ای هنوز پیاده‌سازی نشده است.

---

### `max_ips_to_test`

```toml
max_ips_to_test = 0
```

تعداد IP‌های خوانده‌شده از منبع ورودی را محدود می‌کند. برای نمونه‌گیری از لیست‌های بزرگ یا محدود کردن اسکن‌های سنگین مفید است.

| مقدار | رفتار |
|-------|-------|
| `0` | بدون محدودیت — همه IP‌ها خوانده می‌شوند |
| `N` | حداکثر N عدد IP خوانده می‌شود |

---

### `shuffled`

```toml
shuffled = true
```

ترتیب IP‌های هدف را قبل از شروع اسکن به‌صورت تصادفی درمی‌آورد.

**چرا فعال‌سازی توصیه می‌شود؟**
- از اسکن متوالی یک زیرشبکه یا بلاک شبکه جلوگیری می‌کند
- احتمال فعال‌شدن rate-limiting یا هشدار فایروال را کاهش می‌دهد
- بار اسکن را به‌صورت یکنواخت‌تری روی شبکه توزیع می‌کند

## اجرای پایپ‌لاین

### `pipeline_mode`

```toml
pipeline_mode = "streaming"
```

نحوه اجرای اسکن چند مرحله‌ای را تعریف می‌کند. هر حالت یک تعادل متفاوت بین توان عملیاتی، مصرف حافظه و قابل پیش‌بینی بودن ارائه می‌دهد.

| حالت | توضیح | مناسب برای |
|------|-------|------------|
| `"sequential"` | مراحل یکی پس از دیگری اجرا می‌شوند؛ نتایج بین مراحل روی دیسک ذخیره می‌شوند | محیط‌های کم‌حافظه، اسکن‌های کوچک |
| `"streaming"` | همه مراحل به‌صورت موازی اجرا می‌شوند؛ IP‌ها از طریق کانال‌های بافر شده جریان می‌یابند | بیشترین توان عملیاتی، داده‌های بزرگ |
| `"batch"` | IP‌ها به دسته‌ها تقسیم می‌شوند؛ هر دسته از تمام مراحل عبور می‌کند | هزینه‌های متفاوت مراحل، مصرف حافظه قابل پیش‌بینی |

**جزئیات هر حالت:**

- **`sequential`** — امن‌ترین پروفایل حافظه. هر مرحله صبر می‌کند تا مرحله قبلی کاملاً تمام شود. به دلیل I/O دیسک، کندترین حالت است.

- **`streaming`** *(پیش‌فرض)* — بالاترین کارایی. مراحل از طریق کانال‌های درون‌حافظه‌ای به هم متصل هستند و IP‌ها را به‌صورت همزمان پردازش می‌کنند. ایده‌آل برای اسکن‌های بلادرنگ و حجیم.

- **`batch`** — رویکرد ترکیبی. IP‌ها به دسته‌هایی تقسیم می‌شوند (ببینید [`batch_size`](#batch_size))؛ هر دسته قبل از شروع دسته بعدی از تمام مراحل عبور می‌کند. مصرف حافظه قابل پیش‌بینی‌تر از `streaming`.

## تنظیم دقیق پایپ‌لاین

### `max_ips_per_stage`

```toml
max_ips_per_stage = 100000
```

حداکثر تعداد IP‌هایی که یک مرحله می‌تواند در یک لحظه در حالت زنجیره‌ای نگه دارد.

وقتی این محدودیت به حد خود می‌رسد، مرحله بالادستی **مسدود** می‌شود تا زمانی که فضا آزاد شود این مکانیزم به عنوان یک سقف سخت برای جلوگیری از رشد بی‌حد حافظه عمل می‌کند.

> این یک سقف سخت است، نه یک راهنمای نرم.

---

### `batch_size`

```toml
batch_size = 1000
```

تعداد IP در هر دسته، وقتی `pipeline_mode = "batch"` است.

هر گروه از `batch_size` عدد IP به‌صورت ترتیبی از تمام مراحل پایپ‌لاین عبور می‌کند، قبل از اینکه گروه بعدی شروع شود.

> **فقط زمانی اعمال می‌شود که `pipeline_mode = "batch"` باشد.**

## مثال کامل

<pre dir="ltr">
# ── کنترل اسکن ────────────────────────────────────────────────────────
status_interval   = 1000   # هر ۱ ثانیه وضعیت به UI ارسال شود
stop_after_found  = 100    # پس از ۱۰۰ نتیجه موفق متوقف شود
max_ips_to_test   = 50000  # ورودی به ۵۰ هزار IP محدود شود
shuffled          = true   # ترتیب هدف‌ها تصادفی شود

# ── پایپ‌لاین ─────────────────────────────────────────────────────────
pipeline_mode     = "streaming"
max_ips_per_stage = 100000
batch_size        = 1000
</div>
